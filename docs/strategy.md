# Wear for Weather — 戦略・マネタイズ・ポリシー設計書 v1

> 作成日: 2026-06-01
> 対象: iOS/Android ネイティブ（Web先行の現GAS資産は段階移行）
> 法域: 日本
> マーケット構造: **両面**（コーデ作り手 ↔ 閲覧者）→ **供給フリー原則を不変ルールとして全章適用**
> 収益柱: サブスク（本命）＋ 広告 ＋ アフィリエイト/送客手数料 ＋ 課金アイテム/ポイント
> ステータス: ドラフト（自己批判3ラウンド反映済み。末尾「改訂ログ」参照）

このアプリ固有の構造的前提（全章の土台）:
- **限界費用がある**: 1コーデ再生成 = 5 API呼び出し（コーデJSON gemini-2.5-flash×1 ＋ シーン画像 nano-banana×4）。**2026-06-01に本番 `cost_summary` で実測確認済み**（アバターは既存流用時0回）。ユーザーが叩くほど原価が出る。**「無制限生成」を無料で配ると即赤字**。→ サブスク本命の根拠。
- **身体データを扱う**: プロフィール22項目（身長/体重/体格/骨格/各部位採寸）＋肌の色/顔の形/髪。**要配慮性に近い個人データ**。→ 法務ゲートが重い根拠。
- **毎朝開く理由がある**: 天気連動。**リテンションの天然エンジン**。北極星指標の土台。
- **試着のバイラル性**: 他人のコーデを「自分の体型のアバターで」着られる = 結果画像が固有のシェア素材になる。

## 1. コールドスタート（供給/需要の初期獲得）

両面マーケットの最大の死因は「鶏と卵」。本アプリの強みは **需要側はAIで自給自足できる**こと。閲覧者は他人の投稿がゼロでも「自分用のAIコーデ」が毎朝届くので、**供給ゼロでも単体価値が成立する**。これを最大の武器にする。

### 1-1. 設計の核心: 「片面でも価値が立つ」状態から始める
- **需要側（閲覧者）の初日価値** = AIが今日の天気×自分の体型でコーデ画像を出す。供給投稿が1件も無くても完結。→ **両面の鶏卵問題を「供給は後から乗せる」構造で回避**。
- 供給側（コーデ作り手）は、**閲覧者が育ってから**「自分のコーデが他人のアバターで試着される快感／フォロワー／（後日）送客報酬」を動機に流入させる。

### 1-2. 供給側の初期シード（自社で痩せさせない）
1. **自社シード**: ローンチ前に運営が「お手本コーデ」を体型・天気帯ごとに数十件投入（`discover.html`の初期在庫）。AIアバター試着のショーケースとして機能。
2. **招待スタイリスト枠**: アパレル好き/古着屋/インフルエンサーに**無償の早期アクセス＋公式バッジ**を付与。金銭を払わず「目立てる場所」を提供（供給フリー原則）。
3. **非金銭インセンティブ**: 投稿数・試着され数に応じた**ランキング/バッジ/先行機能解放**。現金は出さない（出すのは成長後＝第4章）。

### 1-3. 需要側の初期獲得
- **招待リファラル**: 既存ユーザーの招待で**両者に生成チケット（消費型）**を付与。※不正対策は第6章（資格到達トリガー・上限・逓減）。
- **天気フックの自然流入**: 「今日の服装どうする」検索/SNS文脈に、結果カードのOG画像で刺さる（第7章）。
- **クローズドβ → 段階開放**: 招待制で需要を絞り、生成原価と品質を測りながら開放幅を制御（原価が読めるまで蛇口を全開にしない）。

### 1-4. コールドスタートの正直な弱点【自己批判反映】
- 供給側の動機が**ローンチ時点では「バッジと自己満足」しかない**（現金報酬は後日）。スタイリスト層が「で、何の得?」となるリスク。→ 緩和策: ①「自分のコーデが他人の体型で再現される」体験自体の新規性を前面に、②早期投稿者に**永久公式バッジ＋将来の送客報酬の優先資格**を確約（言葉だけでなく規約に明記）。
- それでも供給が立たない可能性は残る → **当面は自社シード＋AI供給で需要側を黒字化し、供給は「あれば伸びる」オプション扱い**にしてリスクを下げる（両面を最初から完成させようとしない）。

## 2. ローンチ必須ポリシー（軽くて安全なものだけ）

「お金を外に払い出す機能」は一切入れない（第4章送り）。ローンチは**ユーザーから受け取るだけ**。

### 2-1. 課金プラン3段階（極端回避で中央に誘導）

煽らない。「お得！今だけ！」は使わない。**支払いの痛みを最小化**して淡々と提示。デフォルト選択を本命（中央）に置く。

| プラン | 位置づけ | 主な差分（暫定・第8章で確定） |
|---|---|---|
| **Free（無料）** | 客寄せ・入口 | 1日◯回までの生成、標準画質、広告あり、試着◯回/日 |
| **Standard（本命・デフォルト選択）** | ここに誘導 | 生成回数増、広告なし、高画質、週間プラン解放 |
| **Plus（高アンカー）** | 中央を安く見せる錨 | 生成ほぼ上限緩和、最優先処理、限定シーン/ポーズ |

- **3段階の役割**: 上（Plus）は「中央を安く見せる」ための錨であって主力ではない。下（Free）は供給確保とバイラルの入口。**売りたいのは中央**。デフォルト選択・推奨表示を中央に。
- **身の丈価格**: 大手アパレルアプリや汎用サブスクと**同額帯にしない**。同額だと「信頼の差」で負ける。個人開発/新規である分、**一段安い価格**で「痛みなく試せる」帯に置く（具体額は第8章）。
- **痛みの最小化**: 年額は「月あたり◯円」で併記。価格表示は税込明示（景表法・二重価格表示の禁止に注意【要法務】）。

### 2-2. 供給フリー原則（不変ルール）
- **コーデ作り手から、投稿・公開・試着されることに対して一切課金しない**。供給側機能は常に無料。
- 供給側が払う場面をゼロにする。供給側がカネを出すのは**永久に禁止**（サブスクは「自分が便利に使う」消費者文脈のみ。投稿者が投稿のために課金する設計にしない）。
- 供給側は「払う対象」ではなく「得する場所」: 露出・フォロワー・（後日）送客報酬。

### 2-3. デジタル課金の扱い（ストア）
- サブスク/生成チケット/ポイントは**デジタルコンテンツ → IAP必須**（Apple/Google）。Stripe等の外部決済へ誘導しない（アプリ内表記でのリンク誘導も制約対象）。【ストア審査・実装直前再確認】
- **ポイントは購入不可・換金不可**を厳守（資金決済法の前払式支払手段該当回避＋ストア規約）。→ 詳細は第3章。

### 2-4. 最小限の安全策（ローンチに必須なものだけ）
- **利用規約・プライバシーポリシー**: 身体データ・顔/肌/髪情報の取得目的・AI画像生成への利用・第三者（画像API提供者）への送信を明記【要法務】。
- **未成年**: 年齢確認と未成年の課金同意（民法・消費者契約法）。身体データ取得が未成年の場合の保護者同意【要法務】。
- **AI生成物の権利表示**: 生成画像の権利帰属・商用利用可否・他者の肖像/ブランドロゴ混入時の扱いを規約に。
- **通報導線**: 投稿（供給側）に対する通報を初日から（第6章）。
- **退会/データ削除**: 身体データを含む全データ削除導線（個情法の利用停止・削除請求対応）【要法務】。

### 2-5. このアプリ固有の落とし穴【自己批判反映】
- **「無制限生成」を謳ってはいけない**: 限界費用があるため、Plusでも内部的にフェアユース上限/レート制限を持つ。ただし**「無制限」と表示して実は絞る = 優良誤認（景表法）**。→ 表示は「たっぷり/上限緩和」等、実態と一致する語に。【要法務・景表法】
- **生成失敗時の課金**: 生成チケットを消費したのにAPI失敗で画像が出ない → 返金/再付与ルールを規約とロジック両方に。ローンチ前に必須。

## 3. 補助マネタイズ（横断定義）

### 3-1. 広告
- **無料層のみ**。Standard以上は広告なし（=サブスク誘導圧）。
- 配置: 生成完了待ち（リワード動画で「待ち時間→追加生成チケット」交換が相性良い）、`discover.html`のフィード内ネイティブ広告。
- **UX毀損ライン**: 毎朝の「今日のコーデ」表示そのものに広告を割り込ませない（リテンションの核を汚さない）。広告は「追加で得たい時」に出す。
- リワード広告で得るのは**生成チケット（消費型）**であって現金価値ではない（ストア・資金決済法対応）。

### 3-2. ポイント/チケット経済（厳格設計）
本アプリのポイント = **生成チケット**（コーデ再生成・追加シーン・試着の消費単位）。

- **購入不可・換金不可・有効期限あり**を原則に。これにより:
  - 資金決済法の**前払式支払手段**該当リスクを下げる（ただし「対価性のある前払」になり得る設計は該当しうる → **必ず法務判断**）【要法務・資金決済法】。
  - ストアの「ポイントは購入・換金不可」要件に整合。
- **入手経路**: ①プラン付帯（毎月配布）、②リワード広告、③招待リファラル（第6章の不正対策必須）、④（後日）課金アイテムとして直接購入＝この瞬間に前払式支払手段論点が再燃するので**直接購入の導入は成長後＋法務通過後**。
- **原資管理**: 配布上限・逓減を運営側で制御。配りすぎ=原価爆発なので、**1チケット=API何回相当かを内部原価と紐付けて管理**（第5章コスト計上）。

### 3-3. アフィリエイト/送客手数料（ローンチでは"表示のみ"、決済は後日）
- 提案コーデの各部位（アウター/インナー等）に**実商品リンク（ECアフィリ）**を表示し、送客成果で手数料を得る。
- **ローンチでやってよいのは「外部ECへリンクを貼り、成果報酬を受ける」だけ**（カネが外に出ない＝相対的に軽い）。ただし:
  - **アフィリエイトであることの明示**（ステマ規制・景表法の運用基準）【要法務】。
  - リンク先がアプリ外ブラウザで開くこと、デジタル商品でない物販なのでIAP不要（物理商品の購入はIAP対象外）。ただし**ストアが「外部購入導線」と見なす境界**に注意【ストア審査】。
- **アプリ内決済での物販・出品者への支払い（=カネを外に払い出す）は第4章（成長後）**。

## 4. 成長後ポリシー（現金が動く機能 = 後日）

**「お金を外に払い出す」機能は、法務・KYC・不正対策が固まるまで封印**。順序を厳守。

### 4-1. 供給側への現金還元（送客報酬/投げ銭）— 後日
- コーデ作り手に、**試着→EC送客成果**に応じた現金報酬や、閲覧者からの投げ銭を払う段階。ここで初めて「カネが外に出る」。
- 必要になる重量級対応:
  - **本人確認(KYC)・反社チェック・口座確認**【要法務】
  - **源泉徴収・支払調書・インボイス（適格請求書）対応**（個人クリエイターへの支払い）【要法務・税務】
  - **資金移動業/収納代行の該当性**（プラットフォームがカネを預かり再分配する構造）【要法務・資金決済法】
- **クローバック（求償権）**: 不正取得・チャージバック・規約違反時に**支払済み報酬を取り戻す権利**を規約に。**罰金/違約金とは書かず「不当利得の返還／求償権」の形**で構成【要法務】。

### 4-2. 上位課金の昇格 — 後日
- ローンチのStandard/Plusが安定した後、**プロ向け（スタイリスト分析・一括生成・商用ライセンス）**の上位プランを「昇格」として追加。最初から作らない。

### 4-3. ポイント直接購入 — 後日
- 第3章の通り、チケット直接購入は前払式支払手段論点を抱えるため**法務通過後**。

## 5. 価格戦略の検証方法

### 5-1. 北極星指標（NSM）
候補と選定理由:
- **「週内アクティブ日数（朝の生成完走数）」** を第一候補とする。理由: 天気連動アプリの本質価値は「毎朝開いて今日の服が決まる」習慣化。サブスク継続の先行指標であり、原価（生成回数）とも直結し計測も堅牢。
- 補助指標: 試着完了数、投稿→試着の転換、無料→Standard転換率、生成1回あたり原価。

### 5-2. A/Bテスト項目
- 価格（3段階の絶対額・年/月比率）
- デフォルト選択プラン（中央固定 vs 表示順）
- 無料層の1日生成上限（リテンション vs 原価 vs 転換）
- ペイウォール提示タイミング（生成上限到達時 vs 高品質シーン要求時）
- リワード広告→チケット交換レート

### 5-3. コスト計上（このアプリの肝）
- **1生成あたり実原価 = コーデJSON API + シーン画像API×4** を常時トラッキング。プランの粗利 = サブスク単価 −（配布チケット×平均原価）− ストア手数料(15-30%) − 広告原価相殺。
- **ストア手数料15〜30%を価格に織り込む**（IAP前提）。Web版併存時は手数料差を考慮。
- キャッシュ（気温帯+天気が同じなら再利用、roadmap F6）で**実原価を下げることが粗利改善の主レバー** → 価格より先にキャッシュ率を上げる方が効く可能性。

### 5-4. トレードオフ（正直に）
- 無料層を厚くする = リテンション/バイラル↑ だが原価↑・転換↓。**原価があるため「無料を厚く」は他社より危険**。
- 広告を増やす = 収益↑ だが毎朝体験のUX↓・サブスク転換↓。**核体験には広告を入れない線を死守**。

## 6. 不正対策・通報・還元（濫用防止主眼）

報酬系（招待・紹介・通報・チケット）は**必ず**以下で設計:

### 6-1. 報酬系の鉄則
- **資格到達時トリガー**: 招待報酬は「登録時」でなく「被招待者が実際に生成を◯回完了/◯日継続」した時に発火。登録ボットを無価値化。
- **上限・逓減**: 1ユーザーあたり招待報酬の月間上限、回数で逓減。
- **成果ベース**: 紹介はインストールでなく能動的利用に連動。
- **端末/IP/支払い手段の重複検知**: 自己招待・多重アカウントを検知。
- **報酬原資と罰を連動させない**: 「通報したら報酬」にしない（冤罪インセンティブ回避）。通報は**信頼スコア/対応優先度**で報い、現金にしない。

### 6-2. 通報・モデレーション（両面ゆえ必須）
- 供給側投稿（コーデ画像・キャプション）への通報導線を初日から。
- **AI生成物特有のリスク**: 他者の肖像・ブランドロゴ・不適切表現の混入。生成段階のフィルタ＋事後通報の二段。
- 重要指標（試着数・投稿評価）は**サーバー側記録**に。クライアント改ざん不可に。

### 6-3. クローバック/求償（現金が動く段階で）
- 不正に得た報酬・チャージバック分は**求償権で回収**（罰金表現を避ける、第4章）。【要法務】

## 7. バイラル/シェア設計

「シェアボタンを置く」では不十分。**このアプリ固有のシェアされる理由**を作る。

### 7-1. 固有のシェア素材（優先度順）
1. **【最優先】「私の今日のコーデカード」自動生成**: 自分の体型アバター×今日の天気×コーデ名×ワンポイント を1枚の縦型カード（OG/Instagramストーリー比）に自動合成。毎朝の生成完了時に「シェアする」を自然配置。→ **毎日発生する習慣型シェア**。コストはオンデマンド生成＋キャッシュ（roadmap F6準拠）。
2. **【高】試着ビフォー/アフター**: 他人のコーデを自分のアバターで試着した結果を「元投稿者クレジット付き」でシェア = **シェアが供給側の宣伝にもなる**（両面の好循環）。
3. **【中】週間コーデまとめカード**（`weekly.html`資産を活用）: 1週間のコーデを1枚に。週末シェア需要。
4. **【中】シェア=招待導線の一体化**: シェアカードに招待コード/ディープリンクを埋め、シェア経由インストールを第6章の報酬と接続。

### 7-2. コスト最小化
- シェア画像は**オンデマンド生成＋キャッシュ**。同一条件は再利用。シェアのためだけにAPI再課金しない（既存生成画像の合成で作る）。

### 7-3. 正直な注意【自己批判反映】
- 身体データを使うアプリで「自分の体型アバター」をSNS拡散させるのは**プライバシー懸念**。シェアカードは**体型数値を出さず生成画像のみ**。デフォルトで採寸値・個人特定情報を含めない。共有はオプトイン。

## 付録. プロダクト品質の磨き込み

- **レビュー依頼の出し方**: 星評価依頼は「生成完走・良い結果が出た瞬間」（満足ピーク）にのみ。ストアレビュー誘導のタイミングと頻度を制御（しつこい誘導はストア規約・UX両面でNG）。
- **フィードバック/異議申し立て**: 通報・凍結・チケット消費失敗に対する**異議申し立て窓口**を用意（特に身体データ削除請求は法対応）。
- **行動分析**: NSM（朝の生成完走）を軸にファネル（起動→生成→完走→シェア→招待→転換）を可視化。重要指標はサーバー側。
- **UI/UX土台**: モバイル最優先（roadmap準拠）。毎朝の核体験を最速・最軽量に。ペイウォールは「上限到達時」に痛みなく提示。
